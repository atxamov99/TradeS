# Проверить/перенести после мержа yahyo → main

Контекст: на ветке `yahyo` (github.com/atxamov99/TradeS, коммиты ниже) прогнали баг-фикс аудит
поверх **старого** мобильного приложения (`mobile/src`, AsyncStorage + самодельная очередь
синка) и общего backend. Раз твой mobile-rewrite (TypeScript + Expo Router + WatermelonDB)
полностью заменяет `mobile/src`, часть этих фиксов не перенесётся автоматически при мерже —
их надо проверить/перенести вручную в новую кодовую базу. Некоторые пункты могут уже не
применяться из-за другой архитектуры — это нормально, просто отметь такие как N/A.

Ветка для сверки: `origin/yahyo`, актуальный HEAD `a4bd2e3` (после ребейза на origin/main, 2026-07-17).

## 1. Локализация auth-ошибок (mobile + web)

**Баг:** экраны логина/регистрации показывали сырой текст ошибки с backend (всегда на
узбекском) вместо перевода под текущий язык интерфейса — не было единой точки, которая матчит
`error.response.data.code` на i18n-ключ.

**Что проверить в новом коде:** есть ли у тебя централизованная функция вида
`resolveErrorMessage(error, t)`, которая матчит backend-коды (`ACCOUNT_NOT_FOUND`,
`WRONG_PASSWORD`, `ACCOUNT_EXISTS`, `ACCOUNT_BLOCKED`, `BAG_WEIGHT_REQUIRED`,
`TEST_USER_EXPIRED`, `INSUFFICIENT_STOCK`) на переводы, и используется ли она **везде**, где
ловится auth/product-ошибка (в старом коде именно RegisterScreen забыл её подключить — экран
легко пропустить).

Референс (старый код, для сравнения логики): `mobile/src/utils/apiError.js`,
коммит `60a4784`.

## 2. forgot-password / reset-password: не закодированная ошибка "аккаунт не найден"

**Баг:** `backend/src/services/auth.service.js` — `forgotPassword` и `resetPasswordWithOtp`
кидали 404 без `code` (`ApiError(404, 'Bu raqam bilan hisob topilmadi')`), хотя у login точно
такая же ошибка была закодирована как `ACCOUNT_NOT_FOUND`. Из-за этого на сбросе пароля текст
всегда был на узбекском независимо от языка, плюс на web было **два всплывающих тоста
подряд** — один правильный (от общего axios-интерцептора), второй сырой (страница дублировала
тост вручную).

**Что проверить:** у тебя своя реализация password-reset (phone+email OTP, судя по коммитам
`35288db`/`7b593e4`/`8f67faf`) — она **не** тот же код, но убедись, что там: (а) все
"аккаунт не найден"-подобные ошибки закодированы так же, как у login, (б) нет дублирующихся
toast/alert на одну и ту же ошибку (проверь, есть ли у тебя глобальный error-interceptor, и не
дублирует ли его конкретная страница/экран).

Референс: коммит `6e2aaca` на `yahyo`.

## 3. Кастомный вес мешка (bagWeightKg) — не было UI для произвольного значения

**Баг:** backend всегда принимал любой вес мешка (`Joi.number().min(0)` в
`backend/src/validators/product.validator.js`), но и старый web, и старый mobile ограничивали
выбор пятью пресетами (5/10/25/50/100 кг) — ключ перевода `bagWeightCustom`/`bag_weight_custom`
("Другое"/"Other") существовал во всех локалях, но никуда не был подключён.

**Что проверить:** в новой форме товара (unit = "мешок"/box) есть ли способ ввести
произвольный вес, а не только пресеты.

Референс: коммит `b495820` на `yahyo` (web: `web/src/pages/Products.jsx`, mobile:
`mobile/src/screens/ProductsScreen.js` — для сравнения UX-паттерна).

## 4. Офлайн-синк: тихие сбои никак не показывались пользователю

**Баг:** старая `syncPending()` (`mobile/src/services/syncService.js`) считала
`{synced, failed}`, но ни один вызывающий код это не проверял — ручной тап по бейджу синка на
дашборде не давал вообще никакой обратной связи, успех или провал. Плюс записи, у которых
кончались попытки (attempts >= 5), зависали в статусе `pending` навсегда — исключались из
запроса "что синкать" по фильтру `attempts < 5`, но никогда не помечались `failed`, то есть
были невидимы и для пользователя, и для самой логики синка.

**Что проверить:** WatermelonDB даёт свой sync-протокол из коробки, так что конкретный баг
может не применяться — но сам принцип актуален в любой архитектуре: (а) показывает ли твоё
приложение пользователю, что офлайн-изменения реально не смогли синкнуться (а не просто
крутится спиннер/тихо ничего), (б) есть ли у тебя вообще понятие "запись окончательно не
засинкалась" с явным статусом, а не бесконечные тихие ретраи.

Референс: коммит `1f3835c` на `yahyo`.

## 5. Офлайн-создание товара — дедуп по имени вместо client-id (backend)

**Баг:** `backend/src/services/sync.service.js`, `processSyncBatch` — при повторной отправке
операции "создать товар" (ретрай) сервер проверял дубликат по `name + createdById`. Если
пользователь офлайн создавал **два разных** товара с одинаковым названием (например,
повторный завоз того же товара под тем же названием) — второй молча "склеивался" с первым на
сервере, теряя свою цену/остаток/вес, без единой ошибки.

**Что проверить:** WatermelonDB обычно синкается через свой protocol (`pullChanges`/
`pushChanges`), а не через старый `/sync/push` + `processSyncBatch` — если ты сделал
отдельный sync-эндпоинт под новый протокол, убедись, что идемпотентность там строится на
**стабильном client-id операции**, а не на совпадении полей вроде имени. Если всё ещё идёт
через старый `/sync/push` — этот фикс уже в бэкенде на `yahyo` (см. ниже), просто перенесётся
при мерже.

Референс: коммит `e066b4e` на `yahyo`.

## 8. bagWeightKg — баги персистентности и локализация (mobile)

**Баги (могут не применяться к WatermelonDB-архитектуре, но принцип стоит перепроверить):**
- pull-sync затирал `bagWeightKg` в `null` при подтягивании изменений с сервера, даже когда
  сервер ничего не менял в этом поле (коммит `54e57a1`).
- `productQueries.js` не сохранял `bagWeightKg` при локальном апдейте товара (коммит `8e65e77`).
- `ProductsScreen.js` имел захардкоженные строки вместо i18n-ключей (коммит `e361a78`).

**Что проверить:** в новом `mobile/app`, при редактировании товара с unit="мешок"/box —
сохраняется ли `bagWeightKg` корректно через полный цикл (локально → синк → назад), и
локализованы ли строки экрана товаров.

Референсы: коммиты `54e57a1`, `8e65e77`, `e361a78` на `yahyo`.

## Возможные пересечения при мерже

Ты недавно закинул в `main` `865c11a` ("resolve Critical+High security/correctness issues
across backend, web, mobile") — вероятно, задевает те же файлы backend'а
(`auth.service.js`, `sync.service.js`), что и пункты 2 и 5 выше. При мерже проверь эти два
файла построчно на конфликт логики, а не просто "взять одну из сторон" — фиксы из пунктов 2 и
5 узкие и конкретные, их легко случайно затереть, если авто-merge возьмёт целиком твою версию
файла.
